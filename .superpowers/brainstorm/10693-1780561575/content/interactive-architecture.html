PHI Wall
Mental Health · PHIPA hard constraint
Privacy Layer
Hard rule: Mental Health records are always siloed. They never appear in cross-program aggregates, never flow through the mapping engine to other funder schemas, and are never included in AI M&E metrics unless explicitly scoped to the UW program only.
Why (PHIPA)
Ontario's Personal Health Information Protection Act classifies mental health records as PHI. Consolidation without a proper PHI vault and PIA violates PHIPA. The production roadmap includes a partitioned PHI vault — for the prototype, the 🔒 badge and hard exclusion rule demonstrate awareness.
✅
Consent Engine
Per-program · Cross-program · Dashboard aggregation
Privacy Layer
Three consent levels
1
Program consent — auto-captured at enrollment. "My data is used for [program] reported to [funder]."
2
Cross-program consent — explicit toggle. Required to link a client's records across programs.
3
Dashboard aggregation — implied by cross-program consent. Client appears in org-wide metrics.
Mental Health exception: No consent level permits linking Mental Health to other programs. Hard exclusion.
⛔
File Boundary
Why OneView generates files, not live API pushes
Hard Constraint
The facts
iCARE (IRCC): accepts bulk file upload — ~75% of records already arrive this way. This is the IRCC-endorsed path.

EOIS-CaMS (Employment Ontario): explicitly does not integrate with external systems — IBM Cúram platform, compatibility + privacy constraints. Manual entry only.
Why this is the moat
Every competitor (Salesforce, Apricot, OCMS) also hits this wall. They all leave caseworkers re-keying into funder portals by hand. OneView generates the file. That's the entire value proposition in one sentence.
🔀
LLM Router
Provider-agnostic · One env var to switch
AI M&E Layer
Anti-hallucination guarantee
The LLM never sees raw client data. It only receives pre-computed metric numbers. It writes prose around those numbers. It cannot invent a figure that isn't in the metrics JSON.
Providers
# Switch with one env var LLM_PROVIDER=gemini # free, Google relationship LLM_PROVIDER=claude # highest quality LLM_PROVIDER=groq # fastest inference

Metrics Engine
Deterministic SQL · No LLM involvement
AI M&E Layer
All numbers computed in SQL before the LLM sees anything. The LLM receives a JSON object of pre-computed metrics and writes prose around them. It cannot invent, estimate, or extrapolate.
💾
Report Cache
Demo-safe pre-generation
AI M&E Layer
The showcase report is pre-generated and cached before demo day. Live LLM calls are rate-limited and slow under load. The cache returns instantly. A "Regenerate (live)" toggle is available for judges who want to see real inference.
